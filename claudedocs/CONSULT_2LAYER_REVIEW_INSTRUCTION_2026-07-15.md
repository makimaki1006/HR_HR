## 1. プロダクト背景（前提知識）

- **リポジトリ**: `HR_HR`（Rust / axum）。V2 採用市場ダッシュボード。本番 Render `https://hr-hw.onrender.com`。
- **商談準備レポート とは**: 採用コンサルタントが顧客との商談前に読む「**社内用・顧客配布不可**」の分析資料。顧客に渡す市場レポート（標準/詳細/SP/Ver10）とは**別物・別ルート**（`/consult/brief`）。目的は「面談前に、市場データから言える仮説・示唆・確認すべき質問を整理する」こと。**最終判断はコンサルが面談で行う前提**で、システムは断定しない。
- **モジュール**: `src/handlers/consult/`。仕様の一次ソースは `docs/CONSULT_SUPPORT_PLAN_2026-07-10.md`。
- **データ源**: ①公的統計（e-Stat 等: 将来人口/就業構造/毎月勤労統計/最低賃金/国勢調査OD/経済センサス/住宅土地統計 等）②アップロードされた求人CSV（求人媒体スクレイピング）③企業データベース（従業員数・増減等）。
  - 🔴 **絶対ルール**: 介護データ・自前ハローワークDB（postings/ts_*）は**使わない**。公的統計は由来がHWでも（有効求人倍率等の政府統計は）使ってよい。
- **分析エンジン（既存・今回変更なし）**: 生データ → `evidence`（証拠ID E-001…、種別 OBSERVED/AGGREGATED/PROXY/HYPOTHESIS）→ `4軸判定`（需要/供給/競争/自社給与、High/Medium/Low、**総合点は作らない**）→ `signals`（30個、fired/not-fired）→ `contradictions` → `hypotheses` → `questions` → **`evidence_pack`（構造化JSON）**。
  - 🔴 **設計思想**: 数値計算・集計・閾値判定・信頼度計算は**すべてコード**が確定する。**LLM は「文章化のみ」**、入力は evidence_pack JSON だけ（原データに触れさせない）。
- **LLM**: Google Gemini（モデル `gemini-3.1-flash-lite`、無料枠 15req/分・500req/日）。`GEMINI_API_KEY`（環境変数）未設定時は該当セクションを省略する graceful degradation。API 消費は1レポートあたり最大2コール。

## 2. 今回の変更（レビュー対象）＝ commit `4a24f92`

### 背景（なぜ変えたか）
旧実装の「複合考察」は、シグナルを1つずつ言い換えただけで、読んだコンサルが「**で?結局何?**」となる出来だった。ユーザー（採用コンサル事業者）の要件は次の3つ:
1. **複合示唆と単品示唆の両方**が欲しい（単品＝この数字1つで何が言えるか、複合＝数字を織って何が言えるか）
2. 読んで「**で?結局何?**」が残らないこと（＝すべての示唆が具体的な結論に着地する）
3. **確実に正確に**（データの限界を超えて過剰に読まない。例: 給与が上位でも"条件で勝てている"とは言えない＝休日は未確認）

### 到達目標
`CONSULT_2LAYER_GOLD_STANDARD_v4.md`（同ディレクトリ）。**2層構造**:
- **単品層**: 主要な観測ひとつずつに「観測 → **だから（着地文）**」。顧客の「これどういう意味?」に即答できるよう**全件表示**（情報過多でも網羅を優先、というユーザー判断）。
- **複合層（主役）**: 単品を「複数テーマ織って」戦略示唆にした3〜4本。各々に根拠ID・面談質問・逆の見方（後述の道具箱）。
- **リード仮説**: この1行だけ持ち帰る統括メッセージ。
- **除外**: 使わなかったデータと理由（取りこぼしゼロの明示）。

### 実装の要（この設計が「正確さ」を担保する）
🔑 **正確さはモデルの賢さではなく、渡す"精度ガード（道具）"で決まる** — という設計。実際、無料の小型モデル（flash-lite）でも、下記のガードを与えると**ゴールド水準を安定再現**（後述の実測で6回連続 精度違反0）できることを検証済み。ガードはすべて**コードが pack から動的生成**する（特定地域=大分市の値をハードコードしない＝どの地域/職種でも動く）。

### 変更ファイル（8ファイル、`src/handlers/consult/`）
| ファイル | 役割 |
|---|---|

- **逆証明の道具箱**（`refute_toolbox.rs`）: 複合考察の各項目に、決定的なチェック（T1標本数/T2粒度/T3反対方向シグナル/T4逆因果辞書）で「逆の見方」を機械付与する。LLMに反証させず**コードで反証**する路線。
- **サーバ側検証**（`ai.rs` の validate 層）: LLM出力を信用せず、①根拠IDが実在するか ②禁止表現（`必ず採用できる`/`SalesNow`/`ブリーフ` 等）を含まないか ③個社を主役にしていないか ④（今回追加）overclaim（希少/不足 等）を含まないか、を検査し違反項目を落とす/注記する。

## 3. 🔴 レビューしてほしい観点（本体。ビルド確認ではない）

`CONSULT_2LAYER_GOLD_STANDARD_v4.md` を"あるべき姿"として、実装と実際のAI出力を次の6軸で見てください。

**A. 正確性（最重要）**: 各示唆の「だから」が**データの限界を超えていないか**。
- 給与（客の提示値＝入力）と休日（未確認）を混同して「条件で勝てている」と言っていないか
- **未発火シグナルの内容を断定していないか**（例: タグ「横並び/少ない」・人気「集中」・家賃「高い」・転職層「薄い」・正社員以外「多い」は、対応シグナルが未発火なら言ってはいけない。`theme.rs` の `NOT_FIRED_GUARDS` がこれを動的に禁止する設計）
- 観測限界を事実にすり替えていないか（例: 休日がカードに出ない ≠ 市場に休日が無い）
- 相関を因果と断定していないか

**B. 複合性**: 複合層が**本当に複数テーマを織っているか**。単一シグナルの言い換え（例: 通勤流入の evidence 3つを並べただけ）は複合ではない。異なるテーマ（供給×到達×自社給与 等）を結んで、単一指標には無い結論に着地しているか。

**C. 単品の網羅と着地**: すべての主要観測に「だから」の着地があるか。**孤立データ（どの示唆にも除外にも入らない evidence）がゼロ**か。

**D. 一般化**: 大分市専用になっていないか。`classify()` は metric名で分類しているか（evidence ID や特定数値のハードコードがないか）。`build_accuracy_guards()` は pack の fired 状態で内容が変わるか（別地域・別職種でも破綻しないか）。→ `theme.rs` のテスト `build_guards_works_on_non_oita_synthetic` 等を確認。

**E. 規律**: 断定なし（すべて「〜の可能性」）／因果断定なし／介護・HWデータ不使用／根拠ID実在／禁止語・サービス名（SalesNow/ブリーフ）なし／専門用語・社内語を顧客向け表面に出さない（※商談準備レポートは社内用だが平易な言葉を維持）。

**F. 「で?結局何?」テスト**: レポートを通しで読んで、各示唆・全体に**明確な結論が残るか**。宙に浮く観測・「余地がある」で終わる floater が無いか。

前提: リポジトリ直下。ビルドは **OneDrive 外**の `CARGO_TARGET_DIR` を使う（OneDrive同期がRustビルドを壊すため）。

```bash

#   特に見る: theme:: のテスト（テーマ分類・ガード動的生成・網羅・overclaim検出・非大分合成）

# 2. lint / clippy / fmt
python scripts/lint_statistical_claims.py     # 統計表現lint 0違反
CARGO_TARGET_DIR=/tmp/hr_hr_target cargo clippy --lib   # error 0
CARGO_TARGET_DIR=/tmp/hr_hr_target cargo fmt -- --check

# 3. 顧客向けレポートへの無影響（商談準備レポートは別ルートなので不変のはず）

#   report_basic 7543859a / report_mi e79eca15 / report_extended 7726292e / report_sp 4bba20be が不変か

# 4. 2層描画の目視（合成データでHTML出力する #[ignore] テスト）
CARGO_TARGET_DIR=/tmp/hr_hr_target cargo test --lib dump_two_layer_html_for_visual -- --ignored --nocapture

# 5. 実 Gemini での2層生成（GEMINI_API_KEY があるときのみ）
GEMINI_API_KEY=... CARGO_TARGET_DIR=/tmp/hr_hr_target cargo test --lib <ai.rsのlive #[ignore]テスト> -- --ignored --nocapture
```

### （任意）足場が本当に効くかの独立再現テスト
「精度ガードを与えれば flash-lite でもゴールドを再現できる」という主張を自分で検証したい場合:
1. 本番から商談準備レポートの `evidence_pack.json` を取得（`/consult/evidence_pack.json?session_id=...`、要ログイン）。

   - 実装が本番で使うプロンプト＝ `ai.rs` が組む文字列。`build_accuracy_guards()` の出力を確認するとガードの中身が分かる。
   - APIキーは環境変数で渡すこと（コードやドキュメントにハードコードしない）。

## 5. 既知の残差・妥協点（正直な申告。ここを重点的に見てOK）
1. **文章の切れ味はゴールドの約8割**。flash-lite の「だから」は正確に着地するが、上位LLMほど鋭くない（実務では下書きとして十分、コンサルが仕上げる前提）。
2. **稀に文レベルの言い過ぎ**が混じる（例: 「休日が希少な市場」）。キーワード完全一致ガードをすり抜けるため `overclaim_hits()` の近接検出で対処するが、パターン網羅は完全ではない。→ 新しい過剰表現パターンが漏れていないかは見る価値あり。
3. **本番の実 Gemini 通し確認は未実施**（この実装はまだ本番push前。push後の本番E2Eで「単品全件着地・複合4本・精度違反0」を実データで確認する予定）。レビュー時点でローカルの実Geminiテストを回せるならそれで代替可。
4. ゴールド v4 自体にも軽微な過剰読みが1つあった（転職層 8.8% vs 9.7% を当初「薄い」と書いたが、対応シグナル S-08/S-09 が両方未発火＝正しくは「全国並み」）。**実装のガードはこれを正しく「全国並み」に矯正する**（＝ガード方式が人手より正確、という実証）。この矯正が効いているかは確認ポイント。
