## 1. HW 連携 (hw_enrichment.rs)

### Q1.1 hw_posting_count は市区町村粒度。なぜ posting_change_3m/1y は都道府県粒度か?

**[現象]** 同一テーブル内で hw_posting_count は市区町村単位、posting_change_*m_pct は都道府県単位。同一県内の異なる muni で 3m/1y 値が完全に同じになる。

↓ なぜ?
**[直接原因]** `fetch_pref_posting_changes()` (hw_enrichment.rs:164) は引数に pref のみ取り、`SELECT ... FROM ts_turso_counts WHERE prefecture = ?1` で muni 条件なし。pref キャッシュ後 (line 130-133)、全 muni 行に `pref_changes.get(pref)` の同じ値を流し込む (line 134-142)。

↓ なぜ?
**[構造的原因]** ts_turso_counts テーブル自体が `(snapshot_id, prefecture, emp_group, posting_count)` の 4 列構造で muni 列を持たない。HW 時系列 ETL (`hellowork_etl.py`) で集計レベルが pref で固定された。

↓ なぜ?
**[根本原因]** ETL 設計時に「時系列を 47 都道府県 × 3 雇用形態 × N snapshots」というカラム設計を選び、市区町村別の posting 時系列を保存しないと決めた。これは Turso の row 上限・コスト・ETL 計算量を抑えるためのトレードオフ。

↓ なぜ?
**[真因]** 「市場分析は都道府県粒度で十分」という UX 仮説に基づくが、CSV をアップロードする利用者は **市区町村単位の医療法人/介護施設の採用判断** をしている。粒度仮説が利用者ニーズと食い違う設計初期エラー。

↓ 5 回目の Why
**[究極]** ETL 設計者と UI 設計者の対話不足。Turso 容量制限という「下流の制約」が「上流の値表現」に逆流伝播した結果、市区町村別の動向を見たいユーザーに対して都道府県値を市区町村と詐称表示する事態を招いた。

**逆証明 (Reverse Proof)**:
同一県内の 2 つの muni (例: 千代田区 / 中央区) で `posting_change_3m_pct` が *必ず同一値* になる。ユーザーが「千代田区も中央区も完全に同じ +12.3% だ。怪しい」と気づくか? → 通常の利用者は気づかない。隣接列の `hw_posting_count` は muni ごとに異なるため「データはちゃんと muni 別」と誤信する。
- **既存テストでの検出**: `report_html_qa_test.rs` を grep したが、「同一 pref 内の異なる muni で posting_change が同じ値か」をテストしている箇所なし → **逆証明テスト不在 🔴**
- **必要なテスト**: minimal_agg() に muni 2 件 (千代田区, 中央区) を入れ、同一県でも 3m/1y 値が同じことを検出 (= 詐称している) するテスト。または、ts_turso_counts に muni 列を追加した時に値が異なることを検出するテスト。

**逆因果**:
- 「HW 求人多い地域 → 採用難」 vs 「採用難な地域 → HW 求人増える」: テーブル中段の解釈テキストは「HW 件数が多い muni を参考値」とするのみで因果方向の主張なし。✅ 中立。
- ただし `change_label_3m()` で「大きく増加」と表示すると、ユーザーは「この muni で求人が伸びている = 採用が活発」と読むが、実態は **県単位の動向の希釈**。muni 個別の動きは隠蔽されたまま、ユーザーに「muni 単位の動向を見ている」と誤信させる ⚠️ 構造的逆因果リスク。

**Severity: 🔴 Critical** (P0 #4 で UI 注記したが、データ構造詐称は未解消)

### Q1.2 「+374.3% 大きく増加」のような暴走値が出る原因

**[現象]** ts_turso_counts の初期 snapshot で全国/特定県で異常な値 (+374%、-90% 等) が頻発。

↓ なぜ?
**[直接原因]** `fetch_pref_posting_changes()` で `(latest - prev) / prev * 100.0` を計算する際、`prev` (3m/1y 前) が初期 ETL の不完全データだと小さな分母 → 巨大な変化率。フィルタは `prev > 0.0` のみ (line 187, 192)。

↓ なぜ?
**[構造的原因]** snapshot_id の若い行 (ETL 初期月) は HW スクレイピング範囲が現状より狭く、件数が小さい。これと最新の十分件数を直接除算する。

↓ なぜ?
**[根本原因]** ts_turso_counts は「サンプリングカバレッジが時間的に不安定なまま積み上げられた」時系列。snapshot 間でカバレッジが断絶している。

↓ なぜ?
**[真因]** サンプリング率 (denominator のうち何 % が捕捉できたか) を ts_turso_counts に併記していない。比率 = 件数差 / 件数 で出してしまうため、サンプル数変動を価格変動と区別できない。

↓ 5 回目の Why
**[究極]** 経済指標 (CPI 等) では「base year の固定」+「サンプル数の標準化」が常識だが、HW ETL ではこの手順が抜けた。時系列指標を作るときの基本的な統計設計が省略されている。

**逆証明**:
- 表示が間違っていたら? → 正常な月次変動は通常 ±5〜10% だが、これに該当しない異常値が出ていれば異常。
- 既存対応: report_html/hw_enrichment.rs では UI からテーブル列を削除済 (P0 #4) ✅ + Executive Summary 経由でも値が暴走しないか? → executive_summary.rs では posting_change を直接使わない (給与差分のみ) ので Exec から漏出しない ✅
- ただし integration.rs:584-606 の `render_pct_change_cell()` は依然として呼び出され、別の整合 UI (タブ内の `render_integration`) で表示されている。Section H 削除はテーブル列のみで、`render_integration()` の HW area enrichment セクション table は **残存**。これらに ±100% を超える絶対値を持つときに `None` 化する sanity check 未実装 → **逆証明テスト不在 🔴**

**逆因果**:
- 「求人増加 → 採用ニーズ高い」 vs 「採用ニーズが顕在化 → 求人を増やす」: 両方向あり得るが、+374% という値はそもそも測定誤差なのでどちらの方向の解釈もミスリード。
- 「データ品質が低い → 値が暴れる」 vs 「値が暴れる地域 → データ品質が低い」: これは観察上の双方向で、後者は推論として正しい。ただしレポートで「+374% 増加した地域は活況」と読まれると逆因果。

**Severity: 🔴 Critical** (テーブル列削除のみで、根本的な sanity check 未実装。別経路から漏出する余地あり)

### Q1.3 vacancy_rate_external (e-Stat 由来) の値スケール (0-1 vs 0-100)

**[現象]** `integration.rs:86` で `get_f64(row, "vacancy_rate")` を取得し、`* 100.0` して表示 (line 103)。一方、`HwAreaEnrichment::vacancy_rate_pct` フィールドは既に %単位 (0-100) として定義されている。同じ概念が 2 箇所で異なるスケールで保存される。

↓ なぜ?
**[直接原因]** `report_html/hw_enrichment.rs:63` で fallback_vacancy = `vacancy_rate * 100.0` を計算して `vacancy_rate_pct` (%単位) に流す。一方、enrichment_map から取得した値は `vacancy_rate_pct` ですでに %の想定だが、`hw_enrichment.rs` の本体は `vacancy_rate_pct` を計算する箇所がなく、Default 値の None のみ。

↓ なぜ?
**[構造的原因]** vacancy_rate の生データが Turso に「0.12」(0-1) で保存され、UI 側で「12.0%」(0-100) で表示する変換が分散している。中央集約された normalize 関数が存在しない。

↓ なぜ?
**[根本原因]** e-Stat API (`求人理由が「欠員補充」の比率`) が 0-1 で返ってくるか %で返ってくるかが Turso ETL 時点で曖昧化された。CLAUDE.md L223 では「比率」と表現されているが、scale 単位は明記なし。

↓ なぜ?
**[真因]** データ仕様書で「単位」を必須項目として規定していない。Turso スキーマに `vacancy_rate REAL` とあるだけで、コメントなし。

↓ 5 回目の Why
**[究極]** 「比率」と「率(%)」を日本語で曖昧に交換可能と扱う言語的習慣 + DB スキーマの自己記述性不足 (列コメント・CHECK制約なし)。Rust 型で `Vacancy01(f64)` / `VacancyPct(f64)` のような Newtype 化していれば防げた。

**逆証明**:
- 値が 10倍違っていたら? → 正社員欠員率 12% が 1.2% と表示される (または 120% と表示される)。120% は明らかに異常で気づくが、1.2% は「低くて健全」と誤信される。
- 既存テスト: `report_html_qa_test.rs:1098` で「欠員補充率」のラベルと %プレースホルダの存在チェックはあるが、**値が妥当範囲 (0-100%) にあるかの検証なし** → **逆証明テスト不在 🔴**

**逆因果**:
- 「欠員率高い地域 → 採用難」 vs 「採用難な地域 → 欠員率が高くなる」: これは **同義反復 (定義上同じ)** なので逆因果ではなく定義の循環。
- ただしレポート文「欠員率 30% は要注意」は「欠員率高い → 採用すべき」と読まれるが、実際は「すでに他社も応募が集まらない地域」かもしれない (= 採用困難の自己強化)。逆因果ではないが「示唆の方向が利用者にとって有益でない」可能性。

**Severity: 🔴 Critical** (10倍スケール誤差のリスク、Newtype化されておらずバグ予防策なし)

### Q1.5 enrich_areas の N+1 クエリ回避設計

**[現象]** enrich_areas() (hw_enrichment.rs:82) は muni 数 N に対して、postings は N 回クエリ、ts_turso_counts は pref 数 P 回クエリ (P ≤ 47)。

↓ なぜ?
**[直接原因]** postings は `(prefecture, municipality)` の 2 引数で 1 muni 1 クエリ。ts_turso_counts は `prefecture` 1 引数で集約クエリ。

↓ なぜ?
**[構造的原因]** postings には IN 句一括クエリが使えるが、コードは for ループで個別クエリ (line 101-110)。Turso HTTP API のレイテンシ (US ↔ JP で 50-200ms) を考えると N=30 muni で 1.5〜6 秒の追加遅延。

↓ なぜ?
**[根本原因]** ローカル SQLite (LocalDb) は HTTP 越えではないため per-query レイテンシが μs オーダー。設計者は「ローカルだから for ループでも問題なし」と判断。

↓ なぜ?
**[真因]** これは妥当な判断。ts_turso_counts (Turso) のみが per-query 高コストで、これは pref キャッシュで P 回に削減済 ✅

↓ 5 回目の Why
**[究極]** 設計は妥当。むしろ「postings がローカル SQLite」という前提が変わると (例: HW を Turso 移行)、突然 N+1 問題が顕在化するリスクのみ。

**逆証明**:
- N=100 muni でアップロードした際の所要時間ベンチマークがリポ内になし。
- 既存テスト: enrich_areas に対する単体テストなし → **逆証明テスト不在 🟡**

**逆因果**: 該当なし。

**Severity: 🟢 Info** (現状妥当、将来 HW Turso 化で要再評価)

## 2. SalesNow 統合 (integration.rs / company/fetch.rs)

### Q2.1 地域注目企業の選定基準 (employee_count > 0 + employee_delta_1y IS NOT NULL の論理積)

**[現象]** `fetch.rs:267` の WHERE 句: `employee_count > 0 AND employee_delta_1y IS NOT NULL`

↓ なぜ?
**[直接原因]** 従業員数のあるアクティブ企業 + 過去1年人員変化が記録済の企業のみ抽出。

↓ なぜ?
**[構造的原因]** SalesNow のデータでは employee_count = 0 や delta 欠損行が多数 (新興企業、廃業、データ欠損)。これらを除外しないとレポート末尾の「地域注目企業」が空または不正確になる。

↓ なぜ?
**[根本原因]** 「注目企業」という概念に「データ欠損なし」を含めるのは、UI 表示の安定性確保が目的。

↓ なぜ?

### Q2.2 net_change 計算式 `employee_count * delta / (100 + delta)` の数学的妥当性

**[現象]** SQL: `SUM(employee_count * employee_delta_1y / (100.0 + employee_delta_1y)) as net_change`

↓ なぜ?
**[直接原因]** 「現在従業員数 (= employee_count)」と「過去1年の変化率 (delta%)」から「過去→現在の人員純増減」を逆算するため。

↓ なぜ?
**[構造的原因]** 仮に delta = (現在 - 過去) / 過去 * 100 (一般定義) なら:
  過去 = 現在 / (1 + delta/100) = 現在 * 100 / (100 + delta)
  純増減 = 現在 - 過去 = 現在 * (1 - 100/(100+delta)) = 現在 * delta / (100 + delta) ✅
  → 数式は **delta が「対過去比」**の場合に正しい。

↓ なぜ?
**[根本原因]** SalesNow の `employee_delta_1y` 定義文書がリポ内に存在しない。仮に delta が「対現在比」(= (現在 - 過去) / 現在 * 100) なら式は誤り → `現在 * delta / 100` が正解。

↓ なぜ?
**[真因]** SalesNow API/Turso スキーマに「delta の定義」が明記されておらず、実装者の解釈に依存。`feedback_test_data_validation.md` の「データの中身を検証する」原則に違反。

↓ 5 回目の Why
**[究極]** SalesNow ベンダー仕様の確認 + リポ内 docs への記録 + 計算式の derivation コメント、いずれも欠落。「動いてるからOK」のブラックボックス化。

**逆証明**:
- delta = 100% (1年で2倍) なら式は employee_count * 100 / 200 = 0.5 * employee_count が純増。例: 現在 100 人 → 過去 50 人 → +50 人増。✅ 数学的に正しい (delta=対過去比仮定)。
- delta = -50% なら式は -0.5 * employee_count / 0.5 = -employee_count。例: 現在 100 人 → 過去 200 人 → -100 人。⚠️ -100% より極端。delta=対過去比の典型範囲 [-50%, +∞] に対し、対現在比の範囲は [-∞, +100%]。delta = -50% が対現在比なら過去は 150 人で純増減 -50 (正解)。
- **delta = -50% で式の結果がどちらの仮定でも 50 人減になる偶然** だが、delta = +100% で結果は (対過去比なら) +50 / (対現在比なら) +100 と異なる。
- 既存テスト: net_change の数式検証テストなし → **逆証明テスト不在 🔴**

**逆因果**:
- 「人員増加 → 採用活発」: これは因果として正しい。
- 「採用活発 → 人員増加」: 同義。逆因果ではなく同義。
- ⚠️ ただし「人員減 = 衰退」と読まれるが、子会社化・合併・リストラ後再生など多義性あり。整数値で表示する「年間 +50 人」の文言は強い因果示唆。

**Severity: 🔴 Critical** (定義曖昧で計算誤りの可能性、ベンダー仕様確認必須)

### Q3.6 scatter — 散布図 R² の解釈ラベル

**[現象]** scatter.rs:122-129 で R² > 0.7 = 「強い相関」、> 0.4 = 「中程度の相関」、それ以下 = 「弱い相関」。

↓ なぜ?
**[直接原因]** 一般的な統計教科書の閾値 (Cohen 1988 の慣例) を採用。

↓ なぜ?
**[構造的原因]** R² は決定係数で、相関係数 r の二乗。Cohen の慣例は r 基準で 0.3/0.5/0.8 が weak/medium/strong。R² 換算は 0.09/0.25/0.64。

↓ なぜ?
**[根本原因]** 実装の閾値 0.4 / 0.7 (R²) は r 換算で約 0.63 / 0.84 → Cohen の medium (0.5) より高めの medium、strong (0.8) は妥当。**「中程度の相関」が r=0.63 を意味する** のは Cohen 慣例より厳しい。

↓ なぜ?
**[真因]** R² と r の混同が発生している可能性。仕様作成時に「R² > 0.7 = 強い」と決めたが、これは r=0.84 相当で実は「強い」と呼ぶに値する。「中程度 0.4」は r=0.63 で、これは Cohen 慣例の strong に近い。実装が **過度に conservative** に判定。

↓ 5 回目の Why
**[究極]** 統計学の慣例 (r vs R²) の取り扱いが教科書通りでなく、結果として「実は強い相関を中程度と呼ぶ」誤誘導。

**逆証明**:
- R² = 0.5 のデータ (= r ≈ 0.71) は Cohen 慣例で「強い相関」だが、実装は「中程度」と表示。ユーザーが誤って「相関が弱い → 給与下限と上限は無関係」と判断する逆誘導リスク。
- 既存テスト: R² 解釈ラベルの妥当性テストなし → **逆証明テスト不在 🟡**

**逆因果**:
- 「下限が高い → 上限が高い」: 求人の給与レンジは設計上下限と上限が連動するため、これは因果ではなく **トートロジー**。R² が高いのは「同じ求人の上下限を散布図にしているから」当然。むしろ R² が低いと「給与レンジ設計に一貫性がない」が示唆される。**散布図自体が深い意味を持たない可能性**。

**Severity: 🟡 Warning** (R² 解釈ラベルが過度 conservative、散布図のトートロジー性)

## 逆因果 / 相関≠因果違反検出結果

### 検出された違反候補 (3 件)

#### 1. Q3.8 wage.rs:135 「+10%以上の求人を優先検討すると効率的」
- **誘導**: 「給与高 → 応募集まる → 採用効率」 → これは因果断定
- **逆因果**: 「応募が集まらない求人 → 給与上げ」が実態。給与高 = 高難度職種 = 応募率低 もあり得る
- **既存ヘッジ**: notes §3「相関と因果」あるが、wage 内には明記なし
- **判定**: ⚠️ 違反候補 (利益方向と一致するが、ユーザーには逆因果リスク)

#### 2. Q3.9 seeker.rs:127 「新着求人は市場の最新トレンドを反映」「プラスなら給与水準が上昇傾向」
- **誘導**: 「新着 → 給与上昇」 = 因果
- **逆因果**: 「給与上げる → 再掲載で新着扱い」 / 「campaign 期間 → 一時的に高給与新着」もあり得る
- **既存ヘッジ**: section 内ヘッジなし、notes §3 のみ
- **判定**: ⚠️ 違反候補

#### 3. Q3.10 salesnow.rs:34 「HW 掲載件数が多い法人は採用が活発な傾向」
- **誘導**: 「HW多 → 採用活発」 = 因果。直後に「相関であり、因果は別途検討」とヘッジあり ✅
- **逆因果**: 「採用が困難 → HW にも掲載 → 求人多」(他媒体で集まらないから HW へ)
- **判定**: ⚠️ 文末ヘッジあるが、本文と矛盾。利用者は本文を信じる

## 親セッションへの申し送り Top 5

### 1. 🔴 Q1.1 / Q3.3 dual-location バグ: muni 粒度詐称が Tab UI (`integration.rs`) に残存

P0 #4 で印刷版テーブルから trend 列を削除したが、`integration.rs:499-561` の `render_hw_area_enrichment_section()` (Tab UI 用) では依然として 3 ヶ月推移 / 1 年推移 / 欠員率列を表示。同一県内の muni で値が完全に同一になる詐称が継続。**printed report と Tab UI の修正同期が不徹底**。

**推奨アクション**: integration.rs の同セクションも列削除、または「都道府県値を市区町村行に表示中」のユーザー向け警告を追加。

### 2. 🔴 Q2.2 / Q2.5 SalesNow データの定義未文書化バグ

`employee_delta_1y` の定義 (対過去比 vs 対現在比) と `industry_mapping.confidence` の意味/算出方法がリポ内に文書なし。`net_change` の数式は **delta = 対過去比** を仮定しているが、SalesNow ベンダー仕様確認が抜けている。仮定が違えば全 net_change 値が誤り。

**推奨アクション**:
1. SalesNow ベンダーに `employee_delta_1y` の正式定義を確認
2. リポ docs に記載
3. `feedback_test_data_validation.md` 原則に従い、specific 値で逆証明テスト追加 (例: 100 人 + 100% delta なら +50 人増)

### 3. 🔴 Q3.4 IQR 外れ値除外の片側適用 (notes 詐称)

notes.rs §4 で「給与統計は IQR 適用済」と明記するが、`salary_stats.rs` のヒストグラムは生値 `salary_min_values` を直接渡している。Executive Summary の中央値 (IQR 後) と salary_stats のヒストグラムピーク (IQR 前) が一致せず、注記との不整合 = ユーザー欺瞞。

**推奨アクション**: aggregator.rs で IQR 後の `salary_min_values_iqr` / `salary_max_values_iqr` を新フィールドとして追加し、salary_stats.rs に渡す。または注記を「ヒストグラムは生値、Executive Summary 中央値は IQR 後」と分離記述。

### 4. 🔴 Q4.4 表紙にスコープ注記なし (PDF 部分閲覧での誤認)

表紙 `cover-page` には「ハローワーク」「HW 掲載求人のみ」「全求人市場の代表ではない」のいずれも記載なし。タイトルは「求人市場 総合診断レポート」のみ。PDF 受信者が表紙のみ閲覧した場合、市場全体の診断と完全に誤認する。営業先経営層への誤情報リスクが最大。

**推奨アクション**: cover-page の `cover-target` 直下または `cover-confidential` 内に「※ 本レポートはハローワーク掲載求人 + アップロード CSV を中心とした分析であり、全求人市場の代表ではありません」を追加。既存 QA テスト `p3_spec_9_3_hw_scope_notice_two_places` を「表紙単独でも HW 文字列が含まれること」に強化。

### 5. 🔴 Q1.3 vacancy_rate スケール混乱の Newtype 化必須

0-1 と 0-100 (%) が `integration.rs` と `hw_enrichment.rs::vacancy_rate_pct` で混在。10 倍スケール誤差で「健全」「危機」を反対表示するリスク。Rust の Newtype pattern (`Vacancy01(f64)` / `VacancyPct(f64)`) で型レベル分離していれば防げた。
