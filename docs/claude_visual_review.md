# Claude 視覚レビュー手順

## 目的
cargo test 全 pass + Playwright 自動化だけでは検出できないバグを、LLM の批判的レビューで捕捉する。
特に「合意確認テスト」(テスト作者の誤解とコードの誤解が一致してしまうケース) で本番流出するバグを防ぐ。

## トリガー (実施タイミング)
- PR 作成前 (大きな機能追加)
- 本番デプロイ前
- バグ報告後の同種パターン検査
- 数値ロジック (単位変換・パーセント計算・集計) を含む変更
- レポート HTML/PDF テンプレート変更
- 新しいドメインデータの統合 (e-Stat, SSDSE, Agoop 等)

## 手順

### Step 1: 単体テスト pass
- `cargo test --lib` 全合格 (warning 0 推奨)
- ✅ チェック: 新規ロジックに具体値テスト + ドメイン不変条件アサートあり
- ❌ NG: 「要素が存在する」だけのテスト (`feedback_test_data_validation.md` 参照)

### Step 2: Playwright で全主要セクションのスクリーンショット
媒体分析タブの場合の必須スクリーンショット:
- カバー
- Executive Summary
- 給与統計
- 採用市場逼迫度 (4 軸レーダー + 個別 KPI + データソース折りたたみ)
- 産業ミスマッチ
- 人材デモグラフィック
- 各市区町村カード
- 雇用形態分布
- 給与散布図
- 地域分析 (都道府県/市区町村)
- 最低賃金比較
- 世帯所得 vs 給与
- ライフスタイル特性
- 企業分析
- タグ × 給与
- 求職者心理分析
- 注記・出典・免責

各 section について:
- 画面表示モード (タブ UI)
- 印刷プレビューモード (PDF)
- 両方取得

格納先規約: `docs/screenshots/YYYY-MM-DD/<section>_{view|print}.png`

### Step 3: Claude に視覚レビュー依頼
依頼テンプレート:
```
媒体分析レポートの視覚レビューをお願いします。
スクリーンショット: docs/screenshots/2026-XX-XX/*.png
HTML: /tmp/report_render.html
対象 commit: <SHA>

以下のチェックリストを実行し、発見した問題を報告してください:
[Step 4 のチェックリスト本体を貼り付け]

最後に、Step 5 の同種パターン横展開 grep を実行し、
発見したバグについて他に同じバグがないか確認してください。

報告フォーマットは Step 6 を使用してください。
```

### Step 4: チェックリスト

#### 4-A. 数値妥当性 (ドメイン不変条件)
- [ ] パーセントが 100% を超えていないか (失業率/離職率/構成比 等)
- [ ] 比率が 0-1 範囲内か (誤って 0-100 と混在していないか)
- [ ] 「全国平均」表示が現実的範囲か (例: 失業率 380% = 異常)
- [ ] 「県平均比」「全国比」が 0.01〜10 のオーダーか (それ未満/超過は単位ミス疑い)
- [ ] 給与・支出は正の値、桁数妥当 (月給 0〜100 万円、支出 5〜50 万円)
- [ ] 件数が負にならない
- [ ] 構成比の合計が 100% (±0.5%) に収まる
- [ ] 散布図の点が画面外に飛んでいない

#### 4-B. 内部整合性
- [ ] KPI カードの値と本文中の値が一致
- [ ] レーダーチャートのスコアと軸ラベルの実値が整合
- [ ] テーブル合計と各行の合計が一致 (二重計上検出)
- [ ] グラフのラベルと数値が一致
- [ ] サマリ文の数字と詳細セクションの数字が一致
- [ ] Executive Summary と各セクションのコンクルージョンが矛盾していない

#### 4-C. 表示崩れ
- [ ] チャート空欄/ブランク表示なし (ECharts 初期化確認、`feedback_e2e_chart_verification.md`)
- [ ] 軸スケールが妥当 (極端に圧縮されていない)
- [ ] テキスト切れ・はみ出しなし
- [ ] 印刷プレビューでページ崩れなし
- [ ] PDF 化時に折りたたみが展開されているか
- [ ] 凡例が重ならない/欠けない

#### 4-D. 文言整合性
- [ ] 用語の一貫性 (例: 正社員 vs 正職員、CSV vs 媒体)
- [ ] 因果断定の禁止文言が出ていない (「すべき」「最適」「証明」等)
- [ ] caveat (HW 限定、相関≠因果、粒度制約) の欠落なし
- [ ] 出典・データ取得日の表示
- [ ] 「全求人」と「HW 掲載求人」の混同なし (`feedback_hw_data_scope.md`)

### Step 5: 同種パターン横展開 grep

過去事故から得られた検出 grep。1 件バグを発見したら必ず実行する。

#### パターン A: 二重スケール疑い (* 100.0)
```bash
grep -rn "\* 100\.0\|\* 100\b" src/handlers/survey/report_html/ src/handlers/insight/
```
発見した場合の確認事項:
- 元データ (SQL/fetch) が既にパーセント単位 (0-100) かどうか
- もう一度 100 倍してパーセントに変換していないか
- 「県平均比」のような相対値で 100 倍ミスがないか

#### パターン B: 親子混在の SUM 疑い (.iter().sum で合計)
```bash
grep -rn "iter()\.sum\|\.iter()\.map.*sum" src/handlers/survey/report_html/
```
発見した場合の確認事項:
- 集計対象に「総数」「総支出」「全体」「総計」「消費支出」などの親レコードが含まれていないか
- e-Stat / SSDSE 系では集計行 (AS/AR/CR 等) が含まれることがある
- フィルタで親カテゴリを除外する処理があるか

#### パターン C: find / first / last のフィルタ抜け
```bash
grep -rn "\.find(\|\.first()\|\.last()" src/handlers/survey/report_html/
```
発見した場合の確認事項:
- ソート前提の `.first()` で「集計コード AS/AR/CR」など `A` で始まるコードが先頭に来て混入する可能性
- `.find` の述語が個別カテゴリのみマッチしているか (親カテゴリも誤マッチしていないか)

#### パターン D: 単位混在 (千円 vs 円、月 vs 年)
```bash
grep -rn "1000\|1_000\|10000\|10_000\|12\.0\|/ 12" src/handlers/survey/report_html/
```
発見した場合の確認事項:
- e-Stat の単位 (千円) と HW の単位 (円) を混在していないか
- 年収/月収の換算が二重に適用されていないか

#### パターン E: パーセンテージ計算の分母確認
```bash
grep -rn "/ total\|/ sum\|/ count" src/handlers/survey/report_html/
```
発見した場合の確認事項:
- 分母が「全体合計」か「カテゴリ合計」か明確か
- 分母が 0 の防御 (`if total > 0`) があるか

### Step 6: 報告フォーマット

```
## LLM 視覚レビュー結果

**実施日時**: YYYY-MM-DD HH:MM
**対象**: 媒体分析レポート (commit XXXX)
**スクリーンショット**: docs/screenshots/2026-XX-XX/

### 発見バグ (N 件)
1. [P0/P1/P2] <ファイル:行>: <症状>: <推奨修正>
   - 再現条件: <prefecture/job_type/期間>
   - 期待値: <実値>、表示値: <表示値>

### 確認済み項目
- [x] パーセント上限超過なし
- [x] 親子集計の二重計上なし
- [x] KPI と本文の数値整合
- [x] 全 caveat 文言存在
- [x] 印刷プレビュー崩れなし

### 横展開 grep 結果
- パターン A (* 100.0): N 件、すべて正常 / うち M 件で懸念
- パターン B (.iter().sum): N 件、すべて正常 / うち M 件で親子混在の懸念 → 修正推奨
- パターン C (.find / .first): N 件、すべて正常
- パターン D (単位混在): N 件、すべて正常
- パターン E (分母確認): N 件、すべて正常

### 結論
- ✅ 視覚レビュー pass / ❌ N 件の修正必要
- 修正後再レビュー要 / 不要
```

## メンテナンス
- 新しいバグパターンを発見したら Step 4 のチェックリストと Step 5 の grep に追加
- ユーザーから報告されたバグは必ず横展開 grep で同種を検出
- 半年に 1 回、過去 6 ヶ月のバグ修正コミットを棚卸しし、検出 grep が陳腐化していないか確認
