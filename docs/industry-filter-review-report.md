## 2. 発見事項サマリー

### 2.1 P0: 即時対応（3件）

#### P0-1: `cache.clear()` 全キャッシュ破棄問題
- **発見者**: コーディングプロ
- **場所**: `src/lib.rs` — `set_industry_filter` ハンドラ
- **問題**: フィルタ変更のたびに `state.cache.clear()` で全タブの全キャッシュが消える。6タブ × 多数クエリが毎回再実行され、ユーザー体感で2-5秒の待ち発生
- **対策**: フィルタ関連キーのみの選択的 invalidation、またはTTL自然失効方式に変更
- **影響**: 再クエリ80-90%削減

#### P0-2: アクセシビリティ壊滅的不足
- **発見者**: UI/UXプロ
- **場所**: `templates/dashboard_inline.html` 行37-53, 158-427
- **問題**: ARIA属性なし、キーボード操作不可、`<label>` 未接続。スクリーンリーダー完全不対応。WAI-ARIA 1.2 / WCAG 2.1 AA 未達
- **対策**: `role="tree"`, `aria-expanded`, `tabindex`, キーボードハンドラ追加。`<label>` でチェックボックスをラップ
- **Must項目**: M1(キーボード), M2(ARIA), M3(label接続), M4(モバイル), M5(適用ボタンsticky)

#### P0-3: `renderIndustryTree()` DOM操作の非効率
- **発見者**: コーディングプロ
- **場所**: `templates/dashboard_inline.html` — `renderIndustryTree()` 関数
- **問題**: 524中分類を個別 `appendChild` で描画。大量DOMリフロー発生
- **対策**: DocumentFragment 使用で DOM操作60-70%高速化
- **影響**: ツリー描画時間の大幅短縮

## 3. クロスドメイン分析（複数専門家が合意）

### 3.1 フィルタSQL生成の統一化（DS + コーディング）
5ファイルに散在する産業フィルタSQL生成コードの重複。**単一の共通関数**に統一することで、バグ混入リスク低減 + 保守性向上。

### 3.2 NULL industry_raw の扱い（DS + アナリスト + HR）
ツリー件数とフィルタ結果の不一致（DS）、分析からの脱落（アナリスト）、「その他」の肥大化（HR）はすべて同根。**「未分類」カテゴリの明示的追加**が包括的解決策。

### 3.3 ラベル情報の具体性（アナリスト + UI/UX）
ボタンテキスト「1大分類 + 1中分類」→具体名表示。分析ワークフロー（アナリスト）とユーザビリティ（UI/UX）の双方で改善効果が高い。

## 6. 各専門家レビュー詳細

### 6.1 データサイエンティスト

**重要度: M (Must)**
- M3: industry_tree APIの件数は `industry_raw IS NOT NULL` のみカウント。フィルタ結果はNULL含む全行対象。ユーザーが数値の矛盾に気づく
- M4: `(industry_raw, prefecture)` 複合インデックスが不足。地域×産業の複合フィルタ時にフルスキャン

**重要度: M (Medium)**
- M2: パラメータバインディング方式の混在（positional `?N` vs sequential `?`）。保守リスク

**重要度: S (Suggestion)**
- S2: `build_industry_clause` の統一ラッパー提案。UNION ALL パラメータの簡素化

---

### 6.2 データアナリスト

**高優先度**
- フィルタ状態の可視性不足:「1大分類+1中分類」では具体的に何を選択中か不明
- キャッシュキー不統一: overview のみ municipality 含む構成
- NULL/欠損データの不可視性: industry_raw が空の求人がフィルタ時に完全消失

**中優先度**
- フィルタプリセット保存機能
- CSV出力時のフィルタ条件記載

---

### 6.3 採用コンサルタント
