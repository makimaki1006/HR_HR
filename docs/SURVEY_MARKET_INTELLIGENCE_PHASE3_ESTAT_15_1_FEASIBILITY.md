### 3.2 男女 × 年齢 × 職業 同時取得可否

🟢 **可**。WebFetch メタデータで「男女別：3 項目」「年齢階級：23 区分」「職業大分類：13 区分」が **同一表 15-1 に共存** していることを確認。

期待組み合わせ: 1,965 × 3 × 23 × 13 = **1,762,605 セル** ← e-Stat 画面表記「1,762,605 セル」と完全一致 ⇒ **4 軸全て同時に展開されていることの強い証拠**。

### 3.3 地域コード形式

🟢 **JIS 5 桁** (推定 → 高確度)。

根拠:
- 既存 `municipality_code_master` (`source='estat_commute_od'`、1,917 行) が JIS 5 桁
- 同 e-Stat 国勢調査の通勤 OD (`fetch_commute_od.py`) と表 15-1 は同一統計 (令和2年国勢調査) → **コード体系統一**
- e-Stat API 標準: `cdArea` パラメータは JIS コード直接指定可

### 3.4 不詳・総数・再掲の除外

メタデータで以下が確認済:
- 男女: **総数, 男, 女** (3 項目) → `gender != '総数'` で除外
- 年齢: **再掲項目あり** (23 区分には 15-19, 20-24, ..., 95+, 「総数」、「15-64 (再掲)」等が含まれる) → 数値コードで除外
- 職業: **「分類不能の職業」が明記** (13 区分の最後) → `occupation_code != 'X'` で除外

除外ロジック (実装案):

```python
EXCLUDE_AGE_CODES = {'00000', '00001'}  # 総数 / 再掲
EXCLUDE_OCC_CODES = {'00', 'X'}  # 総数 / 分類不能
EXCLUDE_GENDER_CODES = {'000'}  # 総数
```

正確なコードは API `getMetaInfo` で確定要 (実装時)。

### 3.5 件数規模見積

| 項目 | 値 | 根拠 |
|------|---:|------|
| 表 15-1 全セル数 (e-Stat 表記) | 1,762,605 | WebFetch 実測 |
| 「総数」「再掲」「分類不能」除外後 | 約 **1,200,000〜1,400,000** 行 (推定) | 各軸で「総数」行を除外: 1.76M × (12/13) × (20/23) × (2/3) ≈ 1.0M〜1.2M |
| Phase 0-2 スキーマ (basis 含む) | 1,800 × 21 × 8 × 3 × 2 = **1,814,400** 行 | 既存 `OCCUPATION_POPULATION_FEASIBILITY.md` の期待値 |

**ミスマッチ**:
- e-Stat 15-1: **職業 13 区分** vs スキーマ **職業 21 区分** → **大分類 13 が標準**。スキーマ側を 13 に縮小、または中分類 (約 70 区分) で取得
- e-Stat 15-1: **年齢 23 区分** vs スキーマ **8 階級** → 集計時に 5 歳階級 → 8 階級に集約 (15-24 / 25-34 / .../ 65+ 等)
- e-Stat 15-1: **basis = workplace のみ** → resident は表 12 (年齢欠落) または F2 推定で補完

**Turso 投入見積**:
- 1 行 = 1 write
- フィルタ済み 1,000,000 行 → **1M writes** (Turso 無料枠 10M writes/月 = 10% 消費)
- ローカル SQLite サイズ推定: 1M 行 × 約 80 byte/行 = **80MB**
- Turso 5M 行制限内 (現状 v2_external_* テーブル群と合わせても余裕)

def main():
    # 1. メタ情報取得 (getMetaInfo) → コード辞書
    meta = fetch_meta(STATSDATA_ID)
    occupation_codes = meta['cat02']  # 13 区分
    age_codes        = meta['cat03']  # 23 区分

    # 2. ページング取得 (18 ページ)
    rows = []
    for page in range(18):
        chunk = fetch_data(
            statsDataId=STATSDATA_ID,
            startPosition=page*100000+1,
            limit=100000,
        )
        rows.extend(chunk)

    # 3. フィルタ (総数/再掲/分類不能除外)
    rows = [r for r in rows if not is_excluded(r)]

    # 4. 集計: 5歳階級 → 8 階級
    rows = aggregate_age_groups(rows)

    # 5. SQLite 投入 (basis='workplace' 固定)

## 7. 方針判定: F2 推定モデルとの統合

### 7.1 既存 F2 モデルの位置付け (現状)

`SURVEY_MARKET_INTELLIGENCE_PHASE3_F2_IMPLEMENTATION_PLAN.md`:
- 「市区町村×職業」は推定値 (都道府県職業比 × 市区町村人口按分)
- 表示方針: **比率のみ表示、人数表示NG** (精度 ±20% のため)

### 7.2 表 15-1 取得後の新方針 (推奨)

| basis | データソース | 表示可否 |
|:-----:|------------|:-------:|
| `workplace` (従業地) | 表 15-1 **実測** | 🟢 **人数表示可** |
| `resident` (常住地) | 表 12 (年齢欠落) または F2 推定 | 🟡 **人数表示可だが要注意ラベル** |

→ **F2 モデルは廃止せず、`basis='resident'` 用の補助モデルとして並列保管**。

### 7.3 統合 vs 置換

| 案 | 内容 | 推奨度 |
|----|------|:------:|
| **置換** | F2 を完全削除し表 15-1 のみで運用 | ❌ basis=resident が欠落 |
| **並列保管 (推奨)** | basis 軸で実測 (workplace) と推定 (resident) を併存 | 🟢 |
| 統合 | F2 を「実測値で校正された推定」に再定義 (workplace 実測値で resident 推定の係数を更新) | 🟡 中長期で検討 |

**短期 (1 週間)**: 並列保管で UI に basis 切替トグルを追加
**中期 (1 ヶ月)**: 表 12 + LOD で basis=resident も実測化を検討
