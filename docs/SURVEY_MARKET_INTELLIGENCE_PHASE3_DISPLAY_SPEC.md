## 0. 本書の位置付けと前提

### 0.1 目的

Phase 3 Step 4 で内部承認された **Model F2**（estimate_grade A-）を商品 UI に組み込む際、エンジニアが迷わず実装できるガードレールを提供する。

本書は以下の3点を厳密に定義する:

1. **何を表示するか**（指数/ランク/濃淡の標準形式）
2. **何を表示してはいけないか**（人数表示の Hard NG ルール）
3. **どう表現するか**（注意書き、ラベル、禁止表現）

### 0.2 前提条件

| 項目 | 値 |
|------|-----|
| モデル | Model F2 |
| 内部 estimate_grade | A- |
| weight_source | `hypothesis_v1` (国勢調査実測値による置換予定) |
| データ粒度 | 市区町村 × 大分類職業（08_生産工程 等） |
| 表示単位 | 指数（0-200）、ランク（A/B/C/D）、濃淡（3 段階バー）、順位 |

### 0.3 本書 1 つで実装方針を決められること

本書は、フロントエンド/バックエンドのエンジニアが以下を判断できるように記述する:

- DTO のフィールド名（`thickness_index`, `priority_rank` 等。`population` は禁止）
- テンプレート/JSX の文言（推定バッジ、コンパクト注記）
- API レスポンスのキー名（`*_count`, `*_population`, `*_total_workers` は禁止）
- レポート PDF/HTML のフッター文言

## 1. 商品の主指標と表示形式

商品が顧客に提示する指標は **6 個** に厳格に絞る。それ以外の派生指標を勝手に追加しない。

| # | 指標 | DB 列 | UI 表示形式 | 例 |
|---|------|------|------------|-----|
| 1 | ターゲット厚み指数 | `thickness_index` | 数値 (0-200) + バー | 「ターゲット厚み指数: **142**(推定) ▆▆▆▆▆▆▆▁ (全国平均比 +42%)」 |
| 2 | 配信優先度ランク | `distribution_priority` | A/B/C/D ラベル + バッジ | 「配信優先度: **A ランク** 🟢 (上位 5%)」 |
| 3 | 全国順位 | `rank_in_occupation` | 順位 + 母数 | 「全国 **6 位 / 1,742 市区町村**」 |
| 4 | 採用シナリオ濃淡 | `scenario_conservative_index` / `scenario_standard_index` / `scenario_aggressive_index` | 3 段階バー（相対濃淡） | 「保守 ▆▆ / 標準 ▆▆▆▆▆▆ / 強気 ▆▆▆▆▆▆▆▆▆▆▆」 |
| 5 | 産業集積タイプ | `is_industrial_anchor` | バッジ表示 | 「**🏭 工業集積地**」（anchor=1）/ なし（anchor=0） |
| 6 | 推定グレード | `estimate_grade` | サブ表示（小さく） | 「**検証済み推定** (Model F2 内部 grade A-、weight_source=hypothesis_v1)」 |

### 1.1 指標の表示優先度（情報階層）

UI 上の表示順は **必ず以下の優先順位** で配置する:

1. **第 1 階層**（最初に視線が行く位置）: 配信優先度ランク（A/B/C/D）
2. **第 2 階層**（詳細）: ターゲット厚み指数（0-200）+ バー
3. **第 3 階層**（補足）: 全国順位 + percentile
4. **第 4 階層**（シナリオ詳細）: 保守/標準/強気濃淡
5. **第 5 階層**（注記）: 推定ラベル + データソース + β版バッジ

**人数を匂わせる情報は どの階層にも置かない。**

## 2. 人数表示禁止ルール（Hard NG）

### 2.1 完全禁止表現（修正不可、削除のみ）

| NG 表現 | 理由 |
|---------|------|
| 「○市の生産工程従事者は約 86,000 人」 | 人数推定の絶対精度がない |
| 「○市には◯名の◯◯職」 | 同上 |
| 「採用ターゲット候補総数 ○○人」 | 同上 |
| 「ターゲット母集団 ○○万人」 | 同上 |
| 「保守 467 人 / 標準 1,402 人 / 強気 2,337 人」 | DB に scenario_*_index（指数）のみ保持、人数換算しない |
| 「採用市場規模 ○○億円」 | 人数 × 単価で算出される派生指標も NG |
| 「ターゲット人口 ○○ 名」 | 「人口」「名」が単位として人数を示すため NG |
| 「○○市の生産工程の働き手 ○○人」 | 「働き手」も人数の婉曲表現として NG |
| 「市場規模 ○○ 万人」 | 人数換算の婉曲 NG |
| 「採用候補者プール ○○人」 | 「プール」も同様に人数換算 NG |

### 2.2 グレーゾーン表現（要文脈、部分的 OK）

| 表現 | 条件 |
|------|------|
| 「就業者の○○%」 | **全国構成比**（e-Stat 公表値）を引用する場合のみ OK |
| 「人口○○人の市」 | **市区町村人口**（e-Stat 実測値）を背景情報として引用する場合のみ OK |
| 「労働人口○○万人」 | **国/都道府県粒度**ならば OK（市区町村粒度は NG） |
| 「生産年齢人口比率 ○○%」 | e-Stat 実測値、構成比なら OK |
| 「総就業者○○人」 | e-Stat 実測値、市区町村合計なら OK（個別職業の人数推定は NG） |
| 「○○市は工業都市です」 | `is_industrial_anchor = 1` の場合のみ OK |
| 「○○市は本社集積地です」 | `is_industrial_anchor = 0` AND HQ 過剰判定の場合のみ OK |
| 「採用しやすい」 | 配信優先度 A + 母集団大の場合のみ OK |

**グレーゾーン表現を使う際は、必ずデータ出典を併記すること。**

### 2.3 NG 検出ルール（コードレビュー観点）

レビューワーは以下のキーワードを grep で検出し、コミットを差し戻す:

```regex
(\d+,?\d*)\s*(人|名|名様)
(\d+)\s*万人
(\d+)\s*億円
人口\s*\d+
従事者\s*\d+
ターゲット.*\d+\s*(人|名)
採用候補.*\d+\s*(人|名)
```

例外: 国勢調査の市区町村人口（背景情報）として明示的にラベリングされた場合のみ通す。

## 3. 指数 / ランク / 濃淡の表示推奨

### 3.1 ターゲット厚み指数（thickness_index）

| 項目 | 値 |
|------|-----|
| レンジ | 0 - 200（100 = 全国平均） |
| 配色 | < 80 (薄/灰) / 80-120 (黄) / 120-160 (橙) / > 160 (緑) |
| 単位 | なし（指数） |
| バー表示 | ▆ 8 段階で視覚化 |

**文言例**:
- 「ターゲット厚み指数: **142**(推定) ▆▆▆▆▆▆▆▁ (全国平均比 +42%)」
- 「ターゲット厚み指数: **87**(推定) ▆▆▆▁▁▁▁▁ (全国平均比 -13%)」

**NG**:
- 「○○人」と組み合わせて表示しない
- 絶対値を強調しない（「全国 N 位の指数 142」より「ランク A の指数 142」を優先）

### 3.2 配信優先度ランク（distribution_priority）

| ランク | percentile | 表示文言 |
|--------|-----------|---------|
| A | 上位 5% | 「最優先配信地域」 |
| B | 上位 15% | 「優先配信地域」 |
| C | 上位 50% | 「標準配信地域」 |
| D | それ以下 | 「補助配信地域」 |

**バッジ表示例**:
- 「**A ランク** 🟢 \| 上位 5%」
- 「**B ランク** 🟡 \| 上位 15%」
- 「**C ランク** ⚪ \| 標準」
- 「**D ランク** ⚫ \| 補助」

### 3.3 採用シナリオ濃淡（scenario_*_index）

| シナリオ | DB 列 | 文言 |
|---------|------|------|
| 保守 | `scenario_conservative_index` | 「保守シナリオ: 採用余地 中」 |
| 標準 | `scenario_standard_index` | 「標準シナリオ: 採用余地 高」 |
| 強気 | `scenario_aggressive_index` | 「強気シナリオ: 採用余地 非常に高」 |

**表示形式**: 3 段階バー（保守 / 標準 / 強気）
**数値**: 指数のみ（例: 50 / 150 / 250）← **人数として扱わない**
**定性表現**:
- 指数 < 50: 「低」
- 指数 50-150: 「中」
- 指数 150-250: 「高」
- 指数 > 250: 「非常に高」

**NG**:
- 「保守シナリオで○○人採用可能」
- 「強気シナリオでは ○○ 名の採用が見込める」
- 「保守シナリオで予算 ○○ 円」（金額換算も NG）

### 3.4 産業集積タイプ（is_industrial_anchor）

| 値 | バッジ | 文言 |
|---|--------|------|
| 1 | 🏭 | 「工業集積地」 |
| 0 | （なし） | （表示しない） |

**NG**:
- 「○○市は工場が ○○ 個ある」（個数表示も控える）
- 「製造業就業者○○人」

## 5. 「推定」「参考」ラベル

### 5.1 配置ルール

- **数値の右上**に小さく `(推定)` を必ず付与
- 例: 「ターゲット厚み指数: **142**(推定)」
- 例: 「保守シナリオ指数: 50(推定)」

### 5.2 レポート表紙

- レポート表紙に「**検証済み推定 (estimate_grade A-)**」を明示
- weight_source が `hypothesis_v1` の間は「**β版 (実測値置換予定)**」を補助バッジで表示

### 5.3 weight_source の扱い

| weight_source | 補助バッジ |
|---------------|-----------|
| `hypothesis_v1` | 「**β版 (実測値置換予定)**」 |
| `census_r2_observed`（将来） | 「**正式版**」 |

## 9. 実装エンジニアへのガードレール

### 9.1 フロントエンド（Rust/Axum + Askama テンプレート + HTMX/JS）

**MUST**:
- `thickness_index` をテンプレートに直接埋め込まない（フォーマッタ経由のみ）
- 数値表示時は必ず `(推定)` バッジコンポーネントを併用
- `weight_source = hypothesis_v1` の場合は β 版バッジを必ず表示
- バー表示の段階数は厚み指数の階層に従う（< 80, 80-120, 120-160, > 160）

**MUST NOT**:
- `population` という変数名を使わない（誤解防止）
- `convert_index_to_population()` 等の関数を **作らない**（誘惑禁止）
- 厚み指数を金額換算するコードを書かない
- ハードコードされた人数表記を含むテンプレートを書かない

**推奨コンポーネント名**:

```rust
// Askama template helpers
fn format_thickness_index(idx: i32) -> String { ... }
fn format_priority_rank(rank: &str) -> String { ... }
fn format_scenario_bar(idx: i32) -> String { ... }
fn estimate_badge() -> &'static str { "(推定)" }
fn beta_badge_html() -> &'static str { ... }
```

### 9.2 バックエンド（Rust）

**MUST**:
- DTO field 名は `thickness_index`, `priority_rank`, `scenario_*_index`, `is_industrial_anchor`, `estimate_grade`, `weight_source` 等に統一
- API レスポンスに `weight_source` と `estimate_grade` を必ず含める

**MUST NOT**:
- DTO に `population` フィールドを **追加しない**（DB 列にもなし）
- JSON レスポンスのキー名で `_count`, `_population`, `_total_workers` を **使わない**
- `target_population`, `worker_count`, `employee_estimate` 等の変数名を **使わない**
- 厚み指数から人数を逆算する関数を実装しない

**禁止 DTO 例**:

```rust
// NG
struct OccupationStats {
    population: i32,             // ← 絶対 NG
    target_count: i32,           // ← NG
    estimated_workers: i32,      // ← NG
    market_size_yen: i64,        // ← NG
}
```

**推奨 DTO 例**:

```rust
// OK
struct OccupationStats {
    thickness_index: i32,                  // 0-200
    priority_rank: PriorityRank,           // A/B/C/D
    rank_in_occupation: i32,               // 全国順位
    rank_total: i32,                       // 母数
    scenario_conservative_index: i32,
    scenario_standard_index: i32,
    scenario_aggressive_index: i32,
    is_industrial_anchor: bool,
    estimate_grade: String,                // "A-"
    weight_source: String,                 // "hypothesis_v1"
}
```

### 9.3 SQL クエリ

**MUST NOT**:
- `SELECT population FROM ...`（DB 列名にも `population` を作らない）
- 集計クエリで `SUM(employees) AS total_population`（命名で混入させない）

**OK**:
- `SELECT thickness_index, distribution_priority FROM ...`
- `SUM(weighted_share) AS thickness_index`

### 9.4 ログ出力

**MUST NOT**:
- ログに「○○市の生産工程従事者: ○○人」と出力しない
- 内部デバッグでも人数換算ログは残さない（grep で検出されるリスクあり）

**OK**:
- 「川崎市 08_生産工程: thickness_index=142, priority=A」
