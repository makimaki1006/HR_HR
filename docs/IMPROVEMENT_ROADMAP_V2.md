## Phase 1: 給与分析（2-3週間）

**分類**: 金銭的価値（円で表現される値）の数値分析
**依存**: Phase 0
**実装パターン**: Python事前計算 → SQLiteテーブル → Rust HTMX表示

### 1-1. 適正給与レンジ推定
- 地域×産業×雇用形態で5分位（p10/p25/p50/p75/p90）を事前計算
- Python: `np.percentile` で集計 → `v2_salary_quantiles` テーブル
- 工数: Python 150行

### 1-2. 最低賃金との乖離分析
- 都道府県別最低賃金マスタ（厚労省公開、年次更新）
- 各求人の時給換算 vs 最低賃金 → 乖離率を計算
- 「最低賃金+50円以内」の求人比率 = 処遇改善の余地マップ
- 雇用形態別（パート時給 vs 正社員月給→時給換算）で分析
- `v2_minimum_wage_gap` テーブル
- 工数: Python 100行 + Rust 80行

### 1-3. Shadow Wage Calculator — 影の賃金計算機
**着想**: NBER非賃金アメニティ研究 + Burning Glass Technologies

求人の「見かけの給与」に福利厚生・休日・手当を金銭換算加算し**実質報酬総額**を算出。
- 年間休日差分→時給換算、賞与→月給換算、通勤/住宅手当→月額加算
- 出力: 「表面月給22万 → 実質月給28.5万」
- `v2_shadow_wage` テーブル
- 工数: Python 150行 + Rust 100行

### 1-4. Wage-Amenity Exchange Rate — 賃金-福利厚生交換レート
**着想**: NBERヘドニック賃金回帰（compensating wage differentials理論）

同地域同産業で福利厚生と賃金のトレードオフ関係を回帰分析で推定。
- 「年間休日+1日 ≈ 月給-1,200円」「賞与+1ヶ月 ≈ 月給-8,500円」
- 求人条件の最適組み合わせ設計に直結
- `v2_wage_amenity_rate` テーブル
- 工数: Python 250行 + Rust 100行

### Phase 1 工数合計: Python ~650行 + Rust ~280行

## Phase 2: テキスト分析（2週間）

**分類**: 求人原稿（文字列）の自然言語分析
**依存**: Phase 0
**実装パターン**: Python正規表現/TF-IDF → SQLiteテーブル → Rust HTMX表示

### 2-1. 求人原稿の魅力度スコア
- ルールベース（LLM不使用）: 具体性/ポジティブ表現/構造化度
- 同地域同産業の上位25%をベンチマーク
- 既存テキスト温度計（L-1）の拡張
- `v2_text_attractiveness` テーブル
- 工数: Python 150行 + Rust 100行

### 2-2. Recruitment Signal Decoder — 採用シグナル解読器
**着想**: Indeed Hiring Lab + Eightfold AI

求人テキストのパターンから**採用緊急度・困難度・企業の本音**を4段階分類。
- 🔴緊急（急募/欠員）→ 定着課題
- 🟡緩和（未経験OK/年齢不問）→ 採用難で条件緩和中
- 🟢選別（経験○年以上/有資格者限定）→ 質重視の余裕
- 🔵自信（具体的給与額/賞与実績開示）→ 待遇に自信
- `v2_recruitment_signals` テーブル
- 工数: Python 200行 + Rust 150行

### Phase 2 工数合計: Python ~350行 + Rust ~250行

## Phase 3: 市場構造分析（2-3週間）

**分類**: 地域×産業の集合的パターン分析（個別求人ではなく市場全体）
**依存**: Phase 0
**実装パターン**: Python集計/クラスタリング → SQLiteテーブル → Rust HTMX + ECharts表示

### 3-1. 企業採用戦略の類型化
- 事業所を4象限分類: プレミアム型/給与一本勝負型/福利厚生重視型/コスト優先型
- 給与パーセンタイル × 福利厚生スコアで k=4 クラスタリング
- `v2_employer_strategy` テーブル
- 工数: Python 200行 + Rust 150行

### 3-2. 地域間ベンチマーク
- 2地域選択 → 6軸レーダーチャート自動比較
- 新規テーブル不要（既存v2_*テーブルのクエリ組み合わせ）
- EChartsレーダーチャート
- 工数: Rust 200行 + JS 50行

### 3-3. Employer Monopsony Power Index — 雇用者独占力指数
**着想**: Lightcast社 + 独占禁止経済学のHHI応用

地域×産業での**雇用者側の市場支配力**をHHIで測定。
- 特定企業の求人シェア集中度 → 賃金抑圧リスクの可視化
- 出力: 「高崎市介護: HHI=0.35（寡占）、上位3社で72%占有」
- 既存A層HHI計算の応用（facility_concentration拡張）
- `v2_monopsony_index` テーブル
- 工数: Python 150行 + Rust 100行

### 3-4. Spatial-Conditional Mismatch Detector — 空間的ミスマッチ検出器
**着想**: OECD "Geography of Opportunity" + 空間経済学

求人の地理的分布と条件のミスマッチを検出。
- 通勤圏（30分/60分）内の求人アクセシビリティを算出
- 「高給求人は都市部集中 → 地方の求職者はアクセス不可」の構造問題を可視化
- Leafletヒートマップで「求人砂漠」地帯を表示
- `v2_spatial_mismatch` テーブル
- 工数: Python 300行 + Rust 200行

### Phase 3 工数合計: Python ~650行 + Rust ~650行 + JS ~50行

- 工数: Python 100行 + Rust 50行

### 4-2. 賃金構造基本統計（e-Stat API）
- 都道府県×産業別の公式賃金分布を取得

### 4-4. 最低賃金マスタ（厚労省）

### 現在の限界（10x成長時に致命的）
- DB 1.5GB → Git LFS転送量制限、Render起動時解凍30-60秒
- MemoryStoreセッション → 再起動で全ユーザーログアウト
- format!() HTML生成 → XSSリスク、保守困難

### 推奨方針
- SQLite読み取り専用 + 事前計算テーブル方式は維持（PostgreSQL移行不要）
- 外部データは `compute_v2_external.py` パターンで事前計算→DB格納
- 市区町村コードマッピングテーブル（`municipality_code_map`）が全外部データ統合の基盤
